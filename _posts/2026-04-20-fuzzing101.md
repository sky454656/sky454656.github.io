# Read before start

• AFL uses a non-deterministic testing algorithm, so two fuzzing sessions are never the same. That's why I highly recommend **to set a fixed seed (-s 123)**. This way your fuzzing results will be similar to those shown here and that will allow you to follow the exercises more easily.

# **Meet AFL++**

AFL is a **coverage-guided fuzzer**, which means that it gathers coverage information for each mutated input in order to discover new execution paths and potential bugs. When source code is available AFL can use instrumentation, inserting function calls at the beginning of each basic block (functions, loops, etc.)

커버리지 : 입력이 프로그램 안에서 얼마나 다양한 코드 경로를 실행했는지 나타내는 지표

- **라인 커버리지**: 몇 줄이 실행됐는가
- **함수 커버리지**: 몇 개 함수가 실행됐는가
- **브랜치 커버리지**: `if/else` 같은 분기에서 어느 쪽까지 들어갔는가
- **basic block 커버리지**: 연속해서 실행되는 코드 덩어리들을 얼마나 밟았는가

Instrumentation : 퍼저가 프로그램이 어떤 경로로 어디까지 실행되었는지 알 수 있도록, 컴파일 시점에 코드에 자동으로 삽입되는 추적용 코드

To enable instrumentation for our target application, we need to compile the code with AFL's compilers.

First of all, we're going to clean all previously compiled object files and executables:

```bash
rm -r $HOME/fuzzing_xpdf/install
cd $HOME/fuzzing_xpdf/xpdf-3.02/
make clean
```

And now we're going to build xpdf using the **afl-clang-fast** compiler:

```bash
export LLVM_CONFIG="llvm-config-11"
CC=$HOME/AFLplusplus/afl-clang-fast CXX=$HOME/AFLplusplus/afl-clang-fast++ ./configure --prefix="$HOME/fuzzing_xpdf/install/"
make
make install
```

- `afl-clang-fast`: LLVM 기반 기본 선택지, 무난하고 호환성이 좋음.
- `afl-clang-lto`: 더 빠르고 더 좋은 coverage를 기대할 수 있지만, 빌드 요구사항이 더 까다로울 수 있음

LLVM : 컴파일러·최적화기·코드 생성기 등을 만들 때 쓰는 기반 프레임워크

lexer → parser → IR → optimization → assembly 흐름에서  IR 이후 파트

Now, you can run the fuzzer with the following command:

```bash
afl-fuzz \
-i $HOME/fuzzing_xpdf/pdf_examples/ \
-o $HOME/fuzzing_xpdf/out/ -s 123 \
-- $HOME/fuzzing_xpdf/install/bin/pdftotext @@ $HOME/fuzzing_xpdf/output
```

A brief explanation of each option:

- *i*  : 초기 시드 디렉토리 
- *o :*  퍼징 결과 저장 디렉토리.  / 퍼저의 output
    - `queue/` = **살아남은 corpus**
    - `crashes/` = **크래시 입력**
    - `hangs/` = **타임아웃 입력**
- *s* : 위에서 말한  고정된 랜덤 시드, 123으로 고정
- *@@* : AFL++가 현재 테스트할 입력 파일의 경로로 치환하는 placeholder. 대상 프로그램이 파일 인자를 받는 경우에 사용 
- 프로그램의 결과가 $HOME/fuzzing_xpdf/output에 저장됨

So, basically the fuzzer will run the command

`$HOME/fuzzing_xpdf/install/bin/pdftotext <input-file-name> $HOME/fuzzing_xpdf/output` for each different input file.

지금 fetch 하려는 값 자체가 `2 0 R`

객체 `2 0`의 stream을 처리하는 과정에서 `/Length`를 조회하면, 그 `/Length` 값은 `objRef(2 0 R)`이다. 따라서 `Dict::lookup("Length")`는 `Object::fetch()`를 통해 다시 `XRef::fetch(2,0)`를 호출하게 되고, 현재 처리 중인 동일 객체 `2 0`으로 재진입한다. 이 순환 호출이 반복되면서 무한 재귀가 발생하고, 결국 스택 오버플로우성 크래시로 이어진다.
