For this exercise, we will fuzz **GIMP** image editor. The goal is to find a crash/PoC for [**CVE-2016-4994**](https://www.cvedetails.com/cve/CVE-2016-4994/) in GIMP 2.8.16.

`XCF file`  : GIMP의 기본 프로젝트 파일 형식

GIMP 편집용 원본 프로젝트를 저장한다. 단순히 이미지 픽셀만 있는 것이 아니라 레이어, 채널, 선택 영역 등의 정보가 추가로 들어갈 수 있다.

# Target

**CVE-2016-4994** is an Use-After-Free vulnerability that can be triggered via a crafted XCF file.

Use after free errors occur when a program continues to use a pointer after it has been freed.

This can have any number of adverse consequences, ranging from the corruption of valid data to the execution of arbitrary code.

![[Fuzzing101] Exercise 6 - GIMP](/assets/img/posts/fuzzing101/2026-05-05-fuzzing101-1.png)

` xcf_load_image function in app/xcf/xcf-load.c`  에 취약점이 있다.

# Persistent mode

프로그램을 매번 새로 실행하지 않고, 한 번 실행한 프로세스 안에서 여러 입력을 반복 처리하게 만드는 AFL++ 기능

하나의 프로세스를 계속 사용하면서, 대상 프로그램 안에 퍼징용 코드를 넣고 입력값을 메모리에서 바꿔가며 테스트

# **Do it yourself!**

1. Find an efficient way of modifying GIMP source code to enable AFL persistent mode
1. Create a seed corpus of XCF samples
1. Optional: Create a fuzzing dictionary for the XCF file format
1. Fuzz GIMP until you have a few unique crashes. I recommend you to use as many AFL instances as possible (CPU cores)
1. Triage the crashes to find a PoC for the vulnerability
1. Fix the issues

# Find an efficient way of modifying GIMP source code to enable AFL persistent mode

어디를 어떻게 수정해야 persistent mode가 효율적으로 켜지는지 찾는 것이다.

```bash
int main(...) {
    while (__AFL_LOOP(10000)) {
        gimp 전체 초기화;
        GUI 실행;
        파일 열기;
        종료;
    }
}
```

만약 이런 식으로 반복한다면 GIMP 초기화와 GUI 관련 작업이 매번 반복되기 때문에  persistent mode의 장점이 거의 없고, 비효율적이다.

```bash
GIMP 초기화는 한 번만 수행;

while (__AFL_LOOP(10000)) {
    XCF 파일 로딩;
    이미지/상태 정리;
}
```

이런식으로 하면 효율적으로 여러 입력을 처리할 수 있다. 이러한 위치를 찾는  것이다.

`xcf_load_invoker`  함수의 내부에 다음처럼 패치하면 빠르고, 버그를 찾을 확률이 높다고 한다.

`xcf_load_invoker()`는 XCF 파일 로딩을 시작하는 진입점(wrapper)이고, 그 안에서 파일을 열고 환경 정보를 세팅한 뒤 실제 XCF 파싱 함수로 들어가기 때문에 적합

```bash
GIMP 실행
→ command-line에서 입력 파일 받음
→ 이 파일을 XCF로 판단
→ xcf_load_invoker() 호출
→ filename 가져옴
→ g_fopen(filename, "rb")로 파일 열기
→ XcfInfo 구조체 세팅
→ xcf_load_image() 호출
→ 실제 XCF 파싱/취약 코드 도달
```

# Dictionary 생성

XCF는 XML처럼 문자열 키워드가 많은 포맷이 아니라, 대부분이 바이너리 숫자/offset/길이 값이다. dictionary에는 전체 구조를 다 넣기보다는, 최소한 파서가 XCF로 인식하고 내부 로딩 경로로 들어가게 하는 핵심 문자열만 넣는 경우가 많다고 한다.

```bash
header="gimp xcf "
version_file="file"
version_0="v000"
version_1="v001"
version_2="v002"
version_3="v003"
```

AFL이 헤더를 망가뜨릴 경우, 다시 유효한 입력으로 복구될 가능성을 높이는 장치 정도로 이해하면 되겠다.

```c
g_object_class_install_property (object_class, PROP_WIDTH,
                                   g_param_spec_int ("width", NULL, NULL,
                                                     1, GIMP_MAX_IMAGE_SIZE, 1,
                                                     GIMP_PARAM_READWRITE |
                                                     G_PARAM_CONSTRUCT));
```

`g_param_spec_int()`는 GObject property의 **정수형 속성 범위**를 정의하는 함수<br>
<br>
즉

```bash
property 이름: width
minimum: 1
maximum: GIMP_MAX_IMAGE_SIZE
default: 1
```

이렇게 정의된것이다. `width` 의 최대 사이즈가 `GIMP_MAX_IMAGE_SIZE`  이고, `height`도 똑같으므로 이를 `validation`에 넣어준다.

```bash
if (image_type < GIMP_RGB || image_type > GIMP_INDEXED ||
    width <= 0 || height <= 0 ||

  goto hard_error;
```
