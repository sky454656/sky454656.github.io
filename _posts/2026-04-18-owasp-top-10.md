## 1. OWASP가 무엇인가

OWASP는 Open Worldwide Application Security Project의 약자이며, 소프트웨어 보안을 개선하기 위한 비영리 재단이다. OWASP는 오픈소스 프로젝트, 문서, 표준, 교육 자료, 챕터 커뮤니티, 컨퍼런스 등을 무료·공개 형태로 운영한다. 공식 소개에는 “신뢰할 수 있는 애플리케이션을 구상·개발·구매·운영·유지할 수 있도록 돕는 오픈 커뮤니티”라고 설명되어 있다.

## 2. 가장 최근 발표된 OWASP Top 10 취약점

OWASP Top 10:2025

선정기준은 얼마나 자주 발견되는지,  얼마나 널리 테스트되었는지 , 악용이 쉬운지 , 터졌을 때 영향이 큰지 등을 고려

### A01:2025 Broken Access Control

- 권한 통제가 깨진 상태

- 로그인은 했더라도, 원래 보면 안 되는 다른 사람 정보나 관리자 기능에 접근가능

- URL의 사용자 번호만 바꿨는데 다른 사람 주문 내역이 보이는 경우

- URL 조작, IDOR, API의 POST/PUT/DELETE 접근통제 누락, 권한 상승 등

### A02:2025 Security Misconfiguration

- 설정이 잘못되어 생기는 보안 문제

- 기본 계정이 남아 있거나, 디버그 모드가 켜져 있거나, 불필요한 기능이 열려 있거나, 보안 헤더가 빠져 있는 경우

### A03:2025 Software Supply Chain Failures

- 내 코드가 아니라, 의존 라이브러리·빌드 시스템·배포 과정 등 공급망에서 생기는 보안 문제

- 악성 패키지를 설치했거나, 빌드 파이프라인이 오염되었거나, 배포 산출물이 변조된 경우

### A04:2025 Cryptographic Failures

- 암호화가 잘못되었거나 약해서 중요한 정보가 보호되지 않는 상태입니다.

- 비밀번호를 평문으로 저장하거나, 약한 알고리즘을 쓰거나, 전송 중 암호화가 부실한 경우

### A05:2025 Injection

- 사용자가 넣은 입력이 명령이나 쿼리로 해석되어 시스템이 원래 의도하지 않은 동작을 하는 문제

- SQL Injection, XSS 등

### A06:2025 Insecure Design

- 코드 실수 이전에, 설계 자체가 안전하지 않은 상태

- 원래부터 권한 분리가 제대로 안 되게 설계된 서비스, “비정상 상태를 고려하지 않은 비즈니스 로직

- 위협 모델링, secure design pattern, reference architecture 같은 설계 단계 활동이 중요

### A07:2025 Authentication Failures

- 인증이 약하거나 잘못 구현된 상태

- 약한 비밀번호 정책, 세션 관리 부실, MFA 미적용, 로그인 우회 같은 문제

- MFA : 다중 요소 인증. 비밀번호 + 휴대전화 인증 번호 등

### A08:2025 Software or Data Integrity Failures

- 소프트웨어나 데이터가 신뢰할 수 있는지 검증하지 못하는 문제

- 코드·업데이트 파일·설정 데이터·직렬화 데이터가 변조되었는데도 그대로 실행하거나 신뢰하는 경우

-  신뢰 경계(trust boundaries) 와 무결성 검증 실패에 초점

### A09:2025 Security Logging and Alerting Failures

- 로그를 제대로 남기지 않거나, 남겨도 경고와 대응으로 이어지지 않는 문제

- 침입 시도가 있었는데 기록이 없거나, 기록은 있어도 아무도 알림을 받지 못하는 경우

### A10:2025 Mishandling of Exceptional Conditions

- 예외 상황을 안전하게 처리하지 못해서 생기는 문제

- 입력 누락, 권한 부족, 메모리 부족, 네트워크 오류, 예상치 못한 상태 변화가 생겼을 때 시스템이 안전하게 실패하지 못하고 이상 동작하는 경우

## 3. OWASP Juice Shop


OWASP의 대표 프로젝트 중 하나로, 의도적으로 취약하게 만든 웹 애플리케이션이다. 보안 교육, 인식 제고 데모, CTF, 보안 도구 테스트용으로 사용할 수 있으며, OWASP Top 10 전체와 실제 애플리케이션에서 발견되는 여러 보안 결함을 포함한다. 

이 OWASP Juice Shop을 이용해서 모의 해킹을 실습해보려고 한다.

Burp Suite: 웹사이트와 서버 사이의 HTTP 요청/응답을 관찰·수정·재전송해서 웹 취약점을 찾기 쉽게 해주는 도구

프록시 : 중간에서 대신 전달해 주는 것

내 브라우저 → 프록시 → 웹사이트 서버 구조로 연결되며, Burp Suite는 이 중간 프록시 역할을 수행하여 브라우저가 서버로 보내는 요청과 서버가 브라우저로 반환하는 응답을 가로채어 확인하고, 필요 시 수정·재전송할 수 있도록 한다.

 

- Proxy > Intercept로 이동

- Intercept is off 확인

- Open browser 클릭

- 브라우저 연결

기본 포트가 8080으로 잡혀있는데 왠지 8080 포트가 작동을 안해서 8090 포트로 변경했고, 크롬에서 수동으로 8090 포트로 바꿨더니 크롬 전체를 프록시로 돌리면 모든 웹 요청이 Burp를 거치게 되어서, 설정이 덜 된 상태에서는 일반 웹사이트 접속까지 같이 막힐 수 있다…라는 문제가 생겨서 firfox로 설정했다.

 로컬 loopback 주소(localhost, 127.0.0.1)를 대상으로 한 접속은 Burp를 경유할 때 정상적으로 처리되지 않는 문제가 확인되었다. 이에 따라 Firefox를 Burp 프록시에 연결한 상태에서 로컬 호스트명 대신 시스템의 실제 IPv4 주소를 사용하여 Juice Shop에 접속하였고, 이후 HTTP 요청·응답이 Burp HTTP history에서 정상적으로 수집됨을 확인하였다.
